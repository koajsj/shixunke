汽车租赁系统项目讲解稿

一、项目整体介绍

这个项目是一个汽车租赁管理系统，主要技术栈是：

- 后端：Spring Boot
- 数据库：MySQL
- ORM：Spring Data JPA / Hibernate
- 前端：原生 HTML、CSS、JavaScript
- 运行入口：src/main/java/com/example/demo/DemoApplication.java

项目是一个前后端不分离的 Spring Boot 项目。后端提供 /api/... 接口，前端页面放在 src/main/resources/static 目录下。浏览器访问 http://localhost:8080 时，Spring Boot 会自动返回 index.html 页面。


二、项目启动后发生了什么

运行 DemoApplication 后，Spring Boot 会自动扫描 com.example.demo 包下面的类：

- 扫描 @RestController，生成后端接口。
- 扫描 @Service，创建业务逻辑对象。
- 扫描 JpaRepository，创建数据库访问对象。
- 读取 src/main/resources/application.properties，连接 MySQL。
- 暴露 src/main/resources/static 下的静态页面。

数据库连接配置在 application.properties 里：

spring.datasource.url=jdbc:mysql://localhost:3306/car_rental...
spring.datasource.username=root
spring.datasource.password=12345678
spring.jpa.hibernate.ddl-auto=none

这里 ddl-auto=none 的意思是：JPA 不自动建表，表结构由 src/main/resources/car.sql 控制。这样做是为了和 MySQL 原生表结构保持一致，避免 Hibernate 启动时自动修改表结构。


三、数据库设计

数据库有 4 张核心表：

- Customer：客户表，保存姓名、电话、邮箱、驾驶证号。
- Car：车辆表，保存车牌号、品牌、型号、颜色、年份、车辆状态、日租金。
- Rental：租赁表，记录哪个客户租了哪辆车、租期、金额、订单状态。
- Payment：支付表，记录某个租赁订单的支付信息。

表之间的关系是：

Customer 1 --- N Rental
Car      1 --- N Rental
Rental   1 --- N Payment

也就是说，一个客户可以有多条租赁记录，一辆车可以有多条历史租赁记录，一条租赁记录可以对应支付记录。

数据库里还设置了外键和约束：

- Rental.CustomerID 关联 Customer.CustomerID。
- Rental.CarID 关联 Car.CarID。
- Payment.RentalID 关联 Rental.RentalID。
- ReturnDate 必须大于 RentalDate。
- Payment.Amount 必须大于 0。
- 已完成订单必须填写实际还车日期。

这些约束的作用是防止数据库出现错误数据。


四、后端分层结构

后端采用标准分层：

- Controller：接收浏览器请求，返回响应。
- Service：处理业务逻辑。
- Repository：操作数据库。
- Entity：映射数据库表。

以用户租车为例，完整流程是：

浏览器点击“提交租车申请”
-> app.js 发送 POST /api/rentals/self
-> RentalController 接收请求
-> RentalServiceImpl 处理业务规则
-> RentalRepository / CarRepository 查询和保存数据库
-> 后端返回 JSON
-> 前端刷新页面数据

对应文件：

- 控制层：src/main/java/com/example/demo/controller/RentalController.java
- 业务层：src/main/java/com/example/demo/service/impl/RentalServiceImpl.java
- 数据访问层：src/main/java/com/example/demo/repository/RentalRepository.java
- 实体类：src/main/java/com/example/demo/entity/Rental.java


五、登录、Session 和角色权限

项目里有两个角色：

- 管理员：admin / 123，对应角色 ADMIN。
- 普通用户：user / 123，对应角色 USER。

登录不是用 Spring Security，而是自己写的基础版 Session 登录，核心类有：

- AuthController：处理登录、退出、获取当前登录状态。
- AuthService：校验账号密码，创建登录会话。
- AuthSession：保存当前登录用户的信息。
- AuthInterceptor：拦截接口请求，检查是否登录和是否有权限。
- RequireRole：标记某个接口需要什么角色。

登录流程是：

用户输入账号密码
-> POST /api/auth/login
-> AuthService 判断账号密码
-> 登录成功后创建 AuthSession
-> AuthController 把 AuthSession 存进 HttpSession
-> 浏览器保存 JSESSIONID
-> 后续请求自动带上 JSESSIONID
-> AuthInterceptor 从 Session 中取出角色并判断权限

两个角色就是靠 AuthSession 里的 role 区分的：

- ADMIN 可以访问管理员管理接口。
- USER 只能访问自己的租车、支付、取消订单接口。

前端也会根据登录接口返回的 role 切换页面：

- ADMIN 显示管理员管理台。
- USER 显示用户租车页面。

但真正的权限判断在后端，不靠前端隐藏按钮。前端隐藏按钮只是为了用户体验，后端的 AuthInterceptor 才是真正防止越权访问的地方。


六、WebMvcConfig 的作用

WebMvcConfig 是 Spring MVC 的配置类，位置是：

src/main/java/com/example/demo/config/WebMvcConfig.java

它的作用是把 AuthInterceptor 注册到 Spring MVC 里。

AuthInterceptor 本身只是一个拦截器类，如果不注册，Spring Boot 不知道哪些请求需要经过它。WebMvcConfig 告诉 Spring Boot：所有 /api/** 请求都要先经过 AuthInterceptor 检查。

核心逻辑是：

registry.addInterceptor(authInterceptor)
        .addPathPatterns("/api/**")
        .excludePathPatterns("/api/auth/**");

意思是：

- 拦截 /api/** 下的所有接口。
- 不拦截 /api/auth/**，因为登录、退出、检查登录状态这些接口必须允许未登录访问。

如果没有 WebMvcConfig，AuthInterceptor 不会生效，@RequireRole 权限注解也不会真正被检查。


七、租车业务规则

租赁订单状态有 4 种：

- PENDING_PAYMENT：待支付。
- RENTING：租赁中。
- COMPLETED：已完成。
- CANCELED：已取消。

车辆状态主要有 3 种：

- AVAILABLE：可租。
- RENTED：已出租。
- MAINTENANCE：维护中。

用户租车时，系统会：

1. 检查车辆是否存在。
2. 检查车辆是否可租。
3. 检查还车日期是否晚于取车日期。
4. 自动按“日租金 × 天数”计算金额。
5. 创建 PENDING_PAYMENT 订单。
6. 把车辆状态刷新成 RENTED。

用户点击确认支付时，系统会：

1. 检查订单是不是当前用户自己的。
2. 检查订单状态是否为 PENDING_PAYMENT。
3. 自动生成一条 Payment 支付记录。
4. 把订单状态改成 RENTING。

用户取消订单时，系统会：

1. 检查订单是不是当前用户自己的。
2. 只允许取消未支付订单。
3. 已经有支付记录的订单不能取消。
4. 取消后，如果车辆没有其他待支付或租赁中的订单，车辆恢复为 AVAILABLE。


八、前端实现方式

前端没有使用 Vue、React、Angular 这类框架，而是使用原生 HTML、CSS、JavaScript。

前端文件是：

- src/main/resources/static/index.html：页面结构。
- src/main/resources/static/styles.css：页面样式。
- src/main/resources/static/app.js：页面交互和接口请求。

前端和后端通信使用浏览器自带的 fetch API。

比如登录流程是：

loginForm submit
-> app.js 阻止默认表单提交
-> fetch('/api/auth/login')
-> 登录成功后保存 state.auth
-> 根据 role 显示管理员页面或用户页面

选择原生前端的原因是：这个项目重点是讲清楚 Spring Boot、MySQL、JPA、Controller、Service、Repository 和权限控制。原生前端依赖少，结构直观，不需要额外解释脚手架、组件、路由、打包工具。


九、答辩总结

这个项目从底层看，是一个典型的 Spring Boot MVC 项目。数据库用 MySQL 保存客户、车辆、租赁、支付四类核心数据；后端通过 Controller、Service、Repository、Entity 分层实现业务；前端用原生 HTML、CSS、JavaScript 调用后端 API。

系统支持管理员和普通用户两个角色。管理员负责全量管理，普通用户负责租车、支付和取消订单。租车过程中，系统会校验日期、车辆状态、支付金额和订单归属，保证数据不会乱。


十、老师可能继续细问的问题

1. 为什么要分 Controller、Service、Repository、Entity？

答：为了分工清楚。Controller 接请求，Service 写业务逻辑，Repository 操作数据库，Entity 对应数据库表。这样代码不乱，后期好维护。

2. Entity 和数据库表是怎么对应的？

答：通过 JPA 注解对应。比如 @Entity 表示这是实体类，@Table(name = "Car") 指定数据库表名，@Column(name = "PlateNumber") 指定字段名。

3. Repository 为什么没有实现类？

答：因为 Repository 继承了 JpaRepository，Spring Data JPA 会在运行时自动生成实现类，所以不需要手写实现类。

4. 为什么有 service/interfaces 和 service/impl？

答：接口定义“能做什么”，实现类写“具体怎么做”。小项目可以直接写实现类，但保留接口更符合分层习惯。

5. 登录状态保存在哪里？

答：保存在服务端的 HttpSession 里。浏览器保存的是 JSESSIONID，不是完整用户信息。

6. 为什么登录后要 changeSessionId()？

答：为了防止 Session 固定攻击。登录成功后换一个新的 Session ID，更安全。

7. 为什么普通用户不能看到所有订单？

答：用户接口会从 Session 里拿当前登录用户的 customerId，只查询自己的订单。操作订单时也会校验订单是不是自己的。

8. 为什么车辆状态不是用户手动随便改？

答：车辆是否出租应该由租赁记录决定。如果有待支付或租赁中的订单，车辆就是 RENTED；没有 active 订单时才恢复 AVAILABLE。

9. 为什么用 MySQL，不用 H2？

答：MySQL 更接近真实项目，能演示真实数据库连接、外键、约束、中文编码和持久化数据。

10. 为什么 spring.sql.init.mode=never？

答：正式运行时不希望每次启动都重新建表、覆盖数据，所以设置为 never。需要初始化时手动执行 car.sql。

11. 如果接口出错，前端怎么显示？

答：后端用 ApiControllerAdvice 统一返回错误 JSON，前端 app.js 读取 message 并显示到页面。

12. 如何防止用户输入乱数据？

答：前端有基础限制，后端有 @Valid、实体校验、Service 业务校验，数据库还有外键和 check 约束。真正可靠的是后端和数据库校验。

13. 什么是 ORM？

答：ORM 是对象关系映射。简单说，就是 Java 里的 Car 类对应数据库里的 Car 表，Java 里的对象可以通过 JPA 保存到数据库，不用手写很多 SQL。


十一、Controller、Service、Repository、Entity 和 interfaces 细讲

1. Entity 是什么？

Entity 是数据库表对应的 Java 类。比如数据库里有 Car 表，项目里就有 Car.java。代码里操作的是 Car 对象，JPA 会根据 @Entity、@Table、@Column 这些注解，把 Java 对象和数据库表字段对应起来。

简单说：Entity 负责“把数据库表变成 Java 对象”。

2. Repository 是什么？

Repository 是数据库访问层，负责增删改查。它继承 JpaRepository，所以很多基础方法不用自己写，比如 findAll、findById、save、delete。

项目里有些方法名像 existsByPlateNumber、findByCustomerOrderByRentalIdAsc，Spring Data JPA 会根据方法名自动生成查询逻辑。

简单说：Repository 负责“和数据库打交道”。

3. Service 是什么？

Service 是业务逻辑层。比如租车不是简单保存一条数据，还要判断车辆是否可租、日期是否合法、金额怎么算、车辆状态是否要变成 RENTED、取消订单后车辆是否恢复 AVAILABLE。

这些判断都属于业务规则，所以放在 Service 里，而不是放在 Controller 或 Repository 里。

简单说：Service 负责“业务规则和核心逻辑”。

4. interfaces 和 impl 是什么？

service/interfaces 放接口，service/impl 放实现类。

接口定义“能做什么”，比如 RentalService 里定义创建订单、支付订单、取消订单、查询订单这些方法。

实现类定义“具体怎么做”，比如 RentalServiceImpl 里真正写租车、支付、取消订单的业务逻辑。

简单说：interfaces 负责“规定能力”，impl 负责“具体实现”。

5. Controller 是什么？

Controller 是后端接口入口，负责接收前端请求、接收参数、调用 Service，再把结果返回给前端。

比如前端发送 POST /api/rentals/self，请求会先进 RentalController，然后 Controller 调用 RentalService 创建租车订单。

简单说：Controller 负责“接请求和返回结果”。

6. 一条完整请求链路

以用户租车为例：

前端 app.js
-> POST /api/rentals/self
-> RentalController
-> RentalService 接口
-> RentalServiceImpl 实现类
-> RentalRepository / CarRepository / CustomerRepository
-> MySQL 数据库

返回时再反过来：

MySQL
-> Repository
-> ServiceImpl
-> Controller
-> JSON
-> app.js 渲染页面

总结一句话：

这个项目后端是典型分层结构。Controller 接请求，Service 处理业务，Repository 操作数据库，Entity 映射数据库表，interfaces 定义服务能力，impl 写具体实现。这样每一层职责清楚，代码更好讲，也更好维护。
